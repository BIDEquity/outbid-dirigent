# Generate Conventions

You produce `CONVENTIONS.md` — a focused file that tells any coding agent exactly **how** this codebase writes code. Not what it does (that's ARCHITECTURE.md), but how to write code that fits.

## Why This Matters

Without conventions, an agent writing a new API endpoint will produce generic code that technically works but doesn't match the codebase. With conventions, it writes code that looks like a teammate wrote it — same patterns, same naming, same test approach, same error handling.

This is the difference between a PR that gets approved and one that gets "please follow our patterns" comments.

## Mode

Check `$ARGUMENTS` for mode:
- **Default (generate)**: Create CONVENTIONS.md from scratch
- **`--update`**: Read existing CONVENTIONS.md, verify conventions against current code, update stale sections, preserve `<!-- manual -->` blocks

## Step 1: Discover Conventions

You must **read real code** to find patterns. Never guess. For each category below, find at least 2-3 examples to confirm it's a pattern (not a one-off).

### 1a. Find the dominant patterns

```bash
# What languages are we dealing with?
ls *.py *.rb *.ts *.tsx *.js *.jsx *.go *.rs 2>/dev/null
ls src/ app/ lib/ 2>/dev/null

# Look at the most-edited files — they encode the team's habits
git log --format= --name-only -100 | sort | uniq -c | sort -rn | head -20
```

### 1b. Analyze patterns by category

For each category, search for 3+ examples. If you find fewer than 2 consistent examples, it's not a convention — skip it.

**Authorization & Authentication:**
```bash
# Rails
grep -r "before_action.*authenticate\|load_and_authorize\|current_user\|authorize!" --include="*.rb" -l | head -10
# Node/Python
grep -r "middleware.*auth\|@login_required\|requireAuth\|withAuth" --include="*.ts" --include="*.py" -l | head -10
```

**Data Access Patterns:**
```bash
# Repository/Service pattern?
grep -r "class.*Repository\|class.*Service\|class.*Store" --include="*.rb" --include="*.ts" --include="*.py" -l | head -10
# Direct ORM usage?
grep -r "\.where(\|\.find(\|\.create(\|prisma\.\|Model\." --include="*.rb" --include="*.ts" --include="*.py" -l | head -10
```

**Form/Input Handling:**
```bash
# Rails forms
grep -r "class.*Form\|include ActiveAttr\|include ActiveModel" --include="*.rb" -l | head -10
# React forms
grep -r "useForm\|Formik\|react-hook-form\|zod.*schema" --include="*.tsx" --include="*.ts" -l | head -10
```

**Error Handling:**
```bash
grep -r "rescue\|raise\|catch\|throw\|except\|ApplicationError\|BaseError" --include="*.rb" --include="*.ts" --include="*.py" -l | head -10
```

**Testing Patterns:**
```bash
# Find test files and analyze their structure
ls test/ tests/ spec/ __tests__/ 2>/dev/null
# What test framework?
grep -r "describe\|it \"\|test(\|def test_\|@pytest" --include="*.rb" --include="*.ts" --include="*.py" -l | head -5
# Read 2-3 test files to understand conventions
```

**Naming Conventions:**
```bash
# File naming: kebab-case, snake_case, camelCase?
ls src/**/*.ts 2>/dev/null | head -10
ls app/**/*.rb 2>/dev/null | head -10
# Class/module naming patterns
grep -r "^class \|^module \|^export class\|^export function\|^def " --include="*.rb" --include="*.ts" --include="*.py" | head -20
```

### 1c. Read representative files

Pick 3-5 files that best represent the codebase patterns. Read them fully. Look for:
- Import ordering conventions
- Function/method structure (guard clauses? early returns?)
- Comment style (when comments are used, when they aren't)
- Configuration patterns (env vars? config objects? constants files?)
- Dependency injection vs hard-coded dependencies

## Step 3: Validate

1. **Every file path cited must exist** — verify with ls/glob
2. **Every code example must come from an actual file** — no synthetic examples
3. **Every pattern must have 2+ real occurrences** — one instance is a coincidence, not a convention
4. **Code examples must be verbatim excerpts** (trimmed for length), not paraphrased

## Step 4: Commit (if in Dirigent pipeline)

If running as part of the Dirigent pipeline:
```bash
git add CONVENTIONS.md && git commit -m "docs: generate CONVENTIONS.md"
```

If standalone, do NOT commit.

## Update Mode (--update)

1. Read existing CONVENTIONS.md
2. For each convention, verify it still holds (check 2+ current examples)
3. Remove conventions that are no longer followed
4. Add new conventions discovered in recently changed code
5. Preserve `<!-- manual -->...<!-- /manual -->` blocks
6. Commit: `git add CONVENTIONS.md && git commit -m "docs: update CONVENTIONS.md"`
