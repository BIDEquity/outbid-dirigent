Audit the feature toggle usage in this repository and report compliance gaps.

## Before you begin

Check if `harness-docs/standards-status.md` exists in this repository.
- If it does **not** exist: run `/assess` first to establish a baseline, then return here and continue.
- If it exists: read the rows for `## 06 · Feature Toggles` to understand which toggle-related items are currently failing.

## Instructions

1. **Scan for ad-hoc flag patterns** — look for feature-access logic that bypasses a central toggle service:
   - Environment variable boolean checks used to gate user-facing behaviour: `process.env.FEATURE_X === 'true'`, `os.getenv('FEATURE_X')`, `os.Getenv("FEATURE_X")`
   - Hardcoded boolean constants named with suggestive prefixes: `ENABLE_*`, `FEATURE_*`, `USE_*`, `SHOW_*`
   - Config file booleans (`.yaml`, `.json`, `.toml`) that control user-facing paths but live outside a central toggle registry

2. **Scan the central toggle registry** (if present — look for `features.ts`, `features.py`, `features.go`, or equivalent):
   - Flags missing `owner` field
   - Flags missing `created_date` field
   - Flags missing `planned_removal_date` field
   - Flags where `created_date` is more than 90 days ago and no `planned_removal_date` is set — flag as stale technical debt

3. **Write `harness-docs/feature-toggle-audit.md`:**

   ```markdown
   # Feature Toggle Audit

   _Last run: YYYY-MM-DD. Generated by `/feature-toggle-audit`._

   ## Ad-hoc flag patterns found

   | Location | Pattern | Recommended fix |
   |----------|---------|-----------------|
   | `src/checkout.ts:42` | `process.env.NEW_CHECKOUT === 'true'` | Move to central `features.ts` as `FeatureFlag.NEW_CHECKOUT` |

   ## Central registry gaps

   | Flag | Missing fields | Age (days) | Action |
   |------|---------------|------------|--------|
   | `new-checkout-flow` | `planned_removal_date` | 45 | Add removal date |
   | `old-payment-modal` | `owner`, `planned_removal_date` | 120 | Review — likely stale |

   ## Summary

   - Ad-hoc patterns: N
   - Registry flags missing metadata: N
   - Stale flags (>90 days, no removal date): N
   ```

   If no issues are found in a category, write "No issues found." for that section.

4. After writing the file, output:

   ```
   Feature toggle audit written to docs/feature-toggle-audit.md.
   Found: X ad-hoc patterns · Y registry gaps · Z stale flags

   Next steps:
   - Run /add-feature-toggle to scaffold a central registry if none exists
   - Move ad-hoc patterns to the central registry
   - Set planned_removal_date on stale flags or remove them
   ```

## Update the status file

After writing the audit file, update `harness-docs/standards-status.md`:

1. Find the section heading `## 06 · Feature Toggles`.
2. For each row below, update Verified to today's date, Fixed By to `/feature-toggle-audit`, and update Status and Notes as described:
   - Row matching "Assign every toggle an owner, creation date, and planned removal date": Status → `✅ PASS` if no registry gaps; otherwise leave as ❌ FAIL and set Notes to the gap count.
   - Row matching "Treat toggles active > 90 days without review as technical debt": Status → `✅ PASS` if no stale flags; otherwise ❌ FAIL with Notes = number of stale flags.
3. Recalculate the MUST ✅ and MUST ❌ totals in the Summary table for the `06 · Feature Toggles` row.
