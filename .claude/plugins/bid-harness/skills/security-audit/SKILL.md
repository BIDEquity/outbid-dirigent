Audit this repository for OWASP Top 10 vulnerabilities, hardcoded secrets, and broad IAM permissions using Semgrep.

## Before you begin

Check if `harness-docs/standards-status.md` exists in this repository.
- If it does **not** exist: run `/assess` first to establish a baseline, then return here and continue.
- If it exists: read the rows for `## 08 · Security & Dependency Management` to understand which security items are currently failing.

## Instructions

1. **Semgrep SAST.** Check whether a SAST tool is already configured:
   - Look for a `semgrep` job in CI config files (`.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile`, `.circleci/config.yml`)
   - Look for `.semgrepignore` or `semgrep.yml`
   - Look for `sonar-project.properties` or `.sonarcloud.properties` (SonarQube — skip adding Semgrep if found)

   If **SonarQube is already wired up**: note it in output and skip to step 2.

   If **Semgrep is not yet configured**: add a Semgrep job to the CI pipeline.

   **GitHub Actions — add to `.github/workflows/ci.yml`:**

   ```yaml
   semgrep:
     name: SAST (Semgrep)
     runs-on: ubuntu-latest
     container:
       image: returntocorp/semgrep
     if: github.actor != 'dependabot[bot]'
     steps:
       - uses: actions/checkout@v4
       - name: Run Semgrep OWASP Top 10 + secrets scan
         # semgrep.dev/p/owasp-top-ten: injection, broken auth, sensitive data exposure, etc.
         # semgrep.dev/p/secrets: hardcoded API keys, tokens, credentials
         run: semgrep scan --config=p/owasp-top-ten --config=p/secrets --error .
   ```

   For GitLab CI, Jenkins, CircleCI: add the equivalent job using the `returntocorp/semgrep` Docker image with the same `semgrep scan` command.

   **Run Semgrep locally** and surface findings in the conversation:
   ```bash
   semgrep scan --config=p/owasp-top-ten --config=p/secrets .
   ```
   If `semgrep` is not installed locally, output install instructions (`pip install semgrep` or `brew install semgrep`) and note the CI job was added but local results are unavailable this session.

2. **Hardcoded secrets scan.** Scan source files (exclude `node_modules/`, `.git/`, build output directories) for these patterns (case-insensitive):
   - Variable assignment with sensitive name and string value: `(api_key|apikey|api_secret|password|passwd|token|secret|credential)\s*[=:]\s*["'][^"']{8,}["']`
   - Inline connection strings with credentials: `(postgres|mysql|mongodb|redis)://[^@]+:[^@]+@`

   For each match: report file path, line number, and matched pattern — **redact the actual value**. Flag for human review — do not modify automatically.

3. **IAM / permissions check.** Scan infrastructure config files (`*.tf`, `*.yaml`, `*.yml` in `infra/`, `k8s/`, `helm/`, `.github/workflows/`):
   - Terraform: `"*"` in `actions` blocks, role bindings with `admin` or `owner`
   - Kubernetes: `ClusterRoleBinding` subjects bound to `cluster-admin`, `verbs: ["*"]`
   - GitHub Actions: `permissions: write-all` or workflow-level `permissions: {}` (implicit write-all on older syntax)

   For each finding: report file, line number, and one-sentence explanation of the risk.

4. **Write `harness-docs/security-audit.md`:**

   ```markdown
   # Security Audit

   _Last run: YYYY-MM-DD. Generated by `/security-audit`._

   ## Summary

   | Category | High | Medium | Low |
   |----------|------|--------|-----|
   | Semgrep OWASP | N | N | N |
   | Hardcoded secrets | N | — | — |
   | IAM / permissions | N | N | N |

   ## Semgrep findings

   [Paste Semgrep output table, or "No findings."]

   ## Hardcoded secrets (requires human review)

   | File | Line | Pattern matched | Recommended action |
   |------|------|-----------------|-------------------|
   | `src/db.ts` | 14 | `password =` (value redacted) | Move to environment variable |

   ## IAM / permissions findings

   | File | Line | Finding | Risk |
   |------|------|---------|------|
   | `.github/workflows/deploy.yml` | 4 | `permissions: write-all` | Grants all write permissions; scope to minimum required |
   ```

5. After writing the file, output:

   ```
   Security audit written to harness-docs/security-audit.md.
   High severity: N items

   Address all High items before the next release.
   ```

## Update the status file

After running the security audit, update `harness-docs/standards-status.md`:

1. Find the section heading `## 08 · Security & Dependency Management`.
2. For each row below, update Verified to today's date, Fixed By to `/security-audit`, and update Status and Notes:
   - Row matching "Apply the principle of least privilege to all service accounts, API keys, and IAM roles": Status → `✅ PASS` if no IAM findings; otherwise ❌ FAIL, Notes = finding count.
   - Row matching "Never hardcode secrets, API keys, or credentials": Status → `✅ PASS` if no hardcoded secrets found; otherwise ❌ FAIL, Notes = finding count.
3. Recalculate the MUST ✅ and MUST ❌ totals in the Summary table for the `08 · Security & Dependency Management` row.
