## Area 1: Devbox Capabilities Deep-Dive

### The most important fact: devbox requires nix to be functional

`devbox` is installed at `/usr/local/bin/devbox` (v0.17.0) on this machine, but `devbox info` and `devbox services` fail with a broken nix PATH error. The binary exists but nix is not in PATH. **Detection must be two-step**: `which devbox` is not enough. Always run `devbox version 2>/dev/null` and check exit code. If that fails, skip to tier 2.

### devbox.json structure (confirmed from real schema + source)

```json
{
  "$schema": "https://raw.githubusercontent.com/jetify-com/devbox/0.17.0/.schema/devbox.schema.json",
  "packages": ["nodejs@22", "postgresql@latest", "redis@latest"],
  "env": {
    "PGPORT": "5432",
    "PGHOST": "/tmp/devbox/myapp",
    "REDIS_PORT": "6379"
  },
  "shell": {
    "init_hook": [
      "mkdir -p $PGHOST",
      "initdb 2>/dev/null || true"
    ],
    "scripts": {
      "dev": ["devbox services start", "npm run dev"],
      "test": ["devbox services start", "npm test"]
    }
  }
}
```

**Key schema fields**: `packages` (array or object with version), `env` (flat dict), `shell.init_hook` (array of commands run when entering shell), `shell.scripts` (named multi-step commands). There is **no `services:` block in devbox.json itself** — services are implicit from packages that have plugins.

### How `devbox services` works

Services are driven by **process-compose**, a separate lightweight process manager binary that devbox installs automatically. Each package with a service plugin generates a `process-compose.yaml` in `.devbox/virtenv/<pkg>/`. The plugin definitions live in the devbox binary's embedded nix source:

**postgresql plugin** (from `/nix/store/.../plugins/postgresql/process-compose.yaml`):
```yaml
version: "0.5"
processes:
  postgresql:
    command: "sh -c 'exec postgres -k \"$PGHOST\" -p \"${PGPORT:-5432}\"'"
    shutdown:
      command: "pg_ctl stop -m fast"
    readiness_probe:
      exec:
        command: "pg_isready -p \"${PGPORT:-5432}\""
```
Sets env: `PGDATA=.devbox/virtenv/postgresql/data`, `PGHOST=<virtenv_path>`.

**redis plugin** (from `plugins/redis/process-compose.yaml`):
```yaml
processes:
  redis:
    command: "redis-server $REDIS_CONF --port $REDIS_PORT"
    availability:
      restart: on_failure
      max_restarts: 5
```
Sets env: `REDIS_PORT=6379`, `REDIS_CONF=<devbox_dir>/redis.conf`.

### Can devbox spin up Postgres and Redis natively?

Yes. Both services run as native processes (not Docker). Postgres runs as a unix-socket-bound process via `pg_ctl`. Redis runs as a native redis-server. **No Docker daemon required.** This is the key advantage for CI/headless environments.

**Critical constraint for Postgres**: Before `devbox services start postgresql`, you must run `initdb` to initialize the data directory. The django template shows this pattern:
```json
"test": ["initdb", "devbox services start", "devbox run create_db", "pytest"]
```

### Can `devbox generate` auto-detect from package.json / requirements.txt?

**No.** `devbox generate` produces only: `devcontainer`, `dockerfile`, `direnv`, `readme`, `alias`. None of these detect the stack or generate devbox.json from existing project files. **Dirigent must detect the stack and write devbox.json itself.**

There is **no `devbox init --detect` or `devbox create --auto`**. The `devbox create --template` requires an explicit template name from a fixed list (django, postgres, node-npm, etc.).

### Is devbox viable in CI/headless?

Yes, via `jetify-com/devbox-install-action` GitHub Action (confirmed in devbox's own CI at `.github/workflows/cli-tests.yaml`):
```yaml
- uses: jetify-com/devbox-install-action@v0.14.0
```
This action installs both nix and devbox together. Without it, if the CI runner already has nix installed, devbox can work. On vanilla GitHub-hosted runners, you need the action.

**Headless detection heuristic**:
```python
result = subprocess.run(["devbox", "version"], capture_output=True, timeout=5)
devbox_viable = result.returncode == 0
```

## Area 3: Test Data Seeding Patterns

### Detection heuristic (ordered by confidence)

**High confidence (exact command known):**
1. `prisma/seed.ts` or `prisma/seed.js` exists → command: `npx prisma db seed`
2. `package.json` has `scripts["db:seed"]` or `scripts["seed"]` → use that exact command
3. `db/seeds.rb` exists → Rails project → command: `rails db:seed`
4. `package.json` has `prisma.seed` key → command: `npx prisma db seed`

**Medium confidence (pattern known, path may vary):**
5. `seeds/` directory with `.js`/`.ts` files → likely knex → `npx knex seed:run`
6. `factory_boy` in requirements/pyproject → tests create data via fixtures, no seed command needed, document as "factory-based"
7. Django `fixtures/` dir → `python manage.py loaddata fixtures/initial_data.json` (enumerate files)

**Low confidence / no seed needed:**
8. `conftest.py` uses `sqlite:///:memory:` engine → tests are self-contained, zero infra required
9. `testcontainers-go` in `go.sum` → tests provision their own Docker containers, command is `go test ./...` (requires Docker)
10. Nothing found → document in `INFRA_GAPS.json`, set `seed_command: ""`

### Seed command auto-detection code sketch

```python
def detect_seed_command(repo_path: Path) -> tuple[str, str]:
    """Returns (seed_command, confidence)."""
    # Prisma
    if (repo_path / "prisma" / "seed.ts").exists() or (repo_path / "prisma" / "seed.js").exists():
        return "npx prisma db seed", "high"
    # package.json scripts
    pkg = repo_path / "package.json"
    if pkg.exists():
        scripts = json.loads(pkg.read_text()).get("scripts", {})
        for key in ["db:seed", "seed", "db:seed:run"]:
            if key in scripts:
                return f"npm run {key}", "high"
    # Rails
    if (repo_path / "db" / "seeds.rb").exists():
        return "rails db:seed", "high"
    # Knex
    if (repo_path / "seeds").is_dir():
        return "npx knex seed:run", "medium"
    # Django fixtures
    fixtures = list((repo_path / "fixtures").glob("*.json")) if (repo_path / "fixtures").is_dir() else []
    if fixtures:
        return f"python manage.py loaddata {fixtures[0].name}", "medium"
    return "", "none"
```

## Area 4: CI Config Extraction (Services Tier 3)

### GitHub Actions `services:` block structure

Location: `jobs.<job_id>.services.<service_name>`. Example of a real Postgres + Redis block:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_USER: testuser
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
```

### Reliable extraction algorithm

```python
import yaml, glob

def extract_ci_services(repo_path: Path) -> list[dict]:
    """Extract service definitions from GitHub Actions workflows."""
    services = []
    for wf_path in (repo_path / ".github" / "workflows").glob("*.yml"):
        try:
            data = yaml.safe_load(wf_path.read_text())
        except Exception:
            continue
        for job in (data or {}).get("jobs", {}).values():
            for svc_name, svc in job.get("services", {}).items():
                if not isinstance(svc, dict):
                    continue
                image = svc.get("image", "")
                service_type = _normalize_image(image)  # "postgres:15" -> "postgres"
                ports = svc.get("ports", [])
                host_port = _extract_host_port(ports)  # "5432:5432" -> 5432
                env = svc.get("env", {})
                services.append({
                    "name": svc_name,
                    "type": service_type,
                    "image": image,
                    "host_port": host_port,
                    "env": env,
                })
    return services

def _normalize_image(image: str) -> str:
    name = image.split(":")[0].split("/")[-1]
    return {"postgresql": "postgres", "bitnami-pg": "postgres"}.get(name, name)

def _extract_host_port(ports: list) -> int | None:
    for p in ports:
        parts = str(p).split(":")
        if len(parts) == 2:
            try:
                return int(parts[0])
            except ValueError:
                pass
    return None
```

### Connection string reconstruction from CI env

For the most common services:
- **Postgres**: `postgresql://{POSTGRES_USER:-postgres}:{POSTGRES_PASSWORD}@localhost:{host_port:-5432}/{POSTGRES_DB:-postgres}`
- **Redis**: `redis://localhost:{host_port:-6379}`
- **MySQL**: `mysql://{MYSQL_USER:-root}:{MYSQL_ROOT_PASSWORD}@localhost:{host_port:-3306}/{MYSQL_DATABASE}`

**Caveat on reliability**: CI env values (passwords, usernames) are safe to use for local test runs because they are test credentials. However, CI may have `localhost` hardcoded in a `DATABASE_URL` env var at the job or step level (not in the services block). Also check `env:` at the job level and in individual `steps[*].env` for `DATABASE_URL` overrides.

### Other CI formats

- **GitLab CI**: `services:` block directly under the job, same structure but `alias:` field instead of a named key
- **CircleCI**: `docker:` executor with list of images; second+ images become sidecar services
- **All three** use the same conceptual model: Docker image + env vars + ports
